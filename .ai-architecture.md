# Three.js Clean Architecture — AI Architecture Contract

> **This file is read by both human developers and AI coding agents.**  
> It defines the strict architectural boundaries for this repository.

---

## 1. Layer Map

```
┌─────────────────────────────────────────────────────────┐
│  PRESENTATION  (src/presentation/)                       │
│  DOM overlays, HUD, input handling                       │
│  ← listens to EventBus, never imports infrastructure →  │
├─────────────────────────────────────────────────────────┤
│  USE CASES  (src/use-cases/)                             │
│  AgentBehavior, GameLoop, game mechanics                 │
│  ← pure JS, no Three.js, no DOM APIs →                  │
├─────────────┬───────────────────────────────────────────┤
│  CORE       │  DOMAIN ENTITIES  (src/core/)              │
│  GameEntity │  EventBus, Vector3Utils                    │
│  ← Zero external dependencies →                         │
├─────────────┴───────────────────────────────────────────┤
│  INFRASTRUCTURE  (src/infrastructure/)                   │
│  ├── three/   Scene, Camera, Renderer, RenderLoop        │
│  └── skills/  SkillAdapter, AgentSkillsBridge            │
│  ← may import from all layers above, never the other way │
└─────────────────────────────────────────────────────────┘
```

### Dependency Rule (strict)
- **Outer layers depend on inner layers – never the reverse.**
- `core` has zero imports from other project files.
- `use-cases` may import from `core` only.
- `infrastructure` may import from `use-cases` and `core`.
- `presentation` may import from `use-cases` and `core` (never `infrastructure`).

---

## 2. Three.js Boundary

**Three.js (`import * as THREE`) is ONLY permitted inside `src/infrastructure/three/`.**

Violating this rule:
- Breaks testability of domain logic.
- Couples rendering to business rules.
- Prevents server-side or headless execution of game logic.

**AI Agents: Do not add `import * as THREE` outside the `infrastructure/three` directory.**

---

## 3. Render Loop ↔ Domain Synchronization

```
requestAnimationFrame (RenderLoop.js)
  │
  ├─► GameLoop.tick(delta)          ← USE CASE: domain logic advances
  │     └─► System.update(delta)    ← registered domain systems
  │           └─► EventBus.emit()   ← state changes broadcast
  │
  └─► renderer.render(scene, camera) ← INFRASTRUCTURE: Three.js renders
```

**EventBus** (`src/core/EventBus.js`) is the synchronization mechanism:
- Domain emits events (e.g., `agent:moved`, `agent:spawned`)  
- Infrastructure subscribes and updates Three.js objects accordingly  
- Presentation subscribes and updates DOM accordingly

## 4. AI Agent: Injecting New Features

### Adding a new game mechanic
1. Create a new entity in `src/core/entities/` (no Three.js).
2. Create a use-case in `src/use-cases/` that operates on that entity.
3. Emit events via `eventBus` when state changes.
4. In `src/main.js` (or a dedicated infrastructure binding file), subscribe to those events and update Three.js objects.

### Adding a new AI skill
1. Extend `SkillAdapter` in `src/infrastructure/skills/`.
2. Call `agentBehavior.applySkillResult(agentId, result)` with the response.
3. Domain will emit `agent:skillResult` — subscribe in infrastructure to visualize.

### Reading current domain state
```js
// Available on window._debug in DEV mode
window._debug.agentBehavior.all()   // all live agents
window._debug.gameLoop.elapsedTime  // total elapsed seconds
window._debug.eventBus              // subscribe to any event
```

## 5. Skills Integration

| Platform         | Adapter Class       | Protocol          |
|------------------|---------------------|-------------------|
| agentskills.io   | `HttpSkillAdapter`  | REST POST (JSON)  |
| skills.sh / CLI  | `ShellSkillAdapter` | child_process / stub |

All skill results flow through `AgentBehavior.applySkillResult()` before touching the scene.
